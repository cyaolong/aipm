# 02 PRD

## 需求背景
产品经理在面对新 idea 时，往往需要快速形成可沟通的方案与 PRD，并给出原型以推动协作。现有工具分散、流程割裂，导致效率低、产出不一致。需要一个网页平台将“完善构想 → PRD → 原型”打通，形成闭环。

## 项目定位及发展方向
- 定位：面向产品经理的 AI 方案生成平台，提供结构化 PRD + 低保真原型联动。
- 发展方向：Web MVP 验证后，扩展到 App/小程序；逐步丰富模板与行业化能力。

## 目标与指标（口径）
- 北极星指标：付费转化用户数（当月订阅用户数）。
- 核心指标（1-3 个）：
  - DAU ≥ 50
  - 月收入 ≥ 1000 元 RMB
  - 付费转化率（注册 → 订阅）
- KPI 维度拆解（示例）
  - 增长：注册转化率、新增注册数
  - 转化：免费额度消耗后付费率
  - 留存：7 日/30 日回访率
  - 活跃：人均日使用次数、生成产出次数
  - 收入/成本：ARPU、单次生成成本
- 口径说明：DAU 为当日有登录且产生至少一次生成动作的用户；收入按订阅实际入账计。

## 需求可行性
- 技术可行：基于现有 LLM 与 Figma MCP 接口可实现流程自动化。
- 资源可行：单人、1-2 天开发，MVP 需尽量轻量（登录、项目、订阅、生成）。
- 风险可控：先验证付费意愿与产出价值，再逐步扩展。

## 目标用户与核心场景
- 目标用户：独立/小团队产品经理、创业 PM。
- 核心场景：
  - 输入 idea → 快速得到完善方案
  - 输出标准 PRD → 作为对内外沟通材料
  - 生成原型 → 给设计/开发/业务协作

## MVP 需求范围
- 做：
  - 网页端注册/登录与全流程操作
  - 账号体系与历史项目管理
  - 订阅计费（20 元/月；120 元/年）
  - 初始 token 1000，按调用第三方 API 返回 token 消耗
  - 结构化 PRD（标准深度）
  - Figma 低保真原型链接输出（可复制、可编辑）
- 不做：
  - 移动端 App/小程序
  - 高保真视觉设计
  - 多人协作与权限分级

## 核心流程与功能需求
### 流程闭环
1) 注册/登录 → 进入工作台
2) 创建项目 → 输入 idea
3) 自动完善构想 → 生成 PRD
4) 生成 Figma 原型 → 获取链接
5) 项目历史可回看与复用

### 功能清单
- 用户与权限
  - 邮箱/手机号注册登录
  - 订阅状态与可用 token 展示
- 项目管理
  - 新建项目、历史项目列表、项目详情
  - 保存每次生成的内容（构想/PRD/原型链接）
- 生成引擎
  - 完善构想输出
  - PRD 输出（标准深度，可编辑）
  - 生成 Figma 链接（可复制、可编辑）
- 计费与额度
  - 订阅购买
  - token 额度消耗与提醒

## UI 与交互需求（iOS 组件库）
- Web 平台作为入口，流程一致；原型产出为 iOS 26 风格。
- 表单输入明确，生成流程分步展示（构想 → PRD → 原型）。
- 关键状态需覆盖：加载中、空数据、错误、成功。

## 版本规划与里程碑
- V0（MVP）范围：Web 平台 + 登录 + 项目管理 + 订阅 + token 机制 + 生成三段产出
- V1（扩展）范围：模板增强、行业化版本、协作功能
- 里程碑：
  - D1：完成登录/项目/生成基础流程
  - D2：完成订阅/额度、输出规范与交互状态
- 交付物：PRD、生成样例、Figma 原型链接

## 风险与依赖（可选）
- 风险：单人开发周期紧，可能牺牲体验细节。
- 依赖：第三方 LLM 与 Figma MCP 的稳定性。

## 关键假设与验证计划（可选）
- 付费意愿：通过首批用户验证订阅价格接受度。
- 产出价值：通过用户反馈验证 PRD 与原型可用性。
- token 策略：通过转化漏斗观察免费额度带来的付费转化率。
